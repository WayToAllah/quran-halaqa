# متابعة حفظ القرآن — v2 (إعادة البناء)

⚠️ **هذا فرع تطوير منفصل تماماً عن الموقع الشغال.** الموقع الحالي (`main` → GitHub Pages)
غير متأثر بأي حاجة هنا، ومفيش أي نشر تلقائي من هذا الفرع.

## الحالة الحالية

### المرحلة 1 — الأساس ✅
Vite + TypeScript + Preact + Tailwind CSS v4، وطبقة `src/domain/` (منطق صافٍ 100%)
مع إصلاح مضمّن ومختبر لـ `scoreName(0)`.

### المرحلة 2 — نموذج بيانات Firestore + قواعد الأمان ✅
نموذج multi-tenant (`mosques/{id}/halaqat/{id}/students|records`) بعضوية صريحة،
طبقة `src/data/` كاملة، سكربت هجرة، واختبارات قواعد أمان **مُثبَتة فعلاً عبر CI**.

### المرحلة 3 — الشاشات الأربع ✅ **كاملة**
- ✅ **تسجيل الدخول** — بريد/كلمة سر + فحص عضوية المسجد
- ✅ **الطلاب** — بحث، إضافة/تعديل/حذف بتراجع، نسخ رابط، شارات الترتيب
- ✅ **السجل** — قائمة مُرقّمة (pagination حقيقي)، بحث، حذف بتراجع
- ✅ **الإحصاءات** — فلتر شهري، بطاقات ملخّص، رسم أسبوعي، توزيع تقييمات، لوحتا شرف،
  جدول طلاب بحث/ترتيب
- ✅ **تسجيل** — تاريخ الجلسة، بحث واختيار طالب، بطاقة "ما سمعناه النهارده"
  (بحث مستهدف بـ query مفهرس لسجلات الطالب فقط، مش تحميل الشجرة كاملة)، صفوف لوح/ماضي
  متعددة قابلة للإضافة/الحذف مع تحقق من مدى الآيات الحقيقي لكل سورة، تجويد اختياري
  (نجوم تفاعلية)، وحفظ بنفس منطق الحماية من التطبيق الحي (منع الحفظ المزدوج،
  تأكيد قبل حفظ جلسة فارغة، إعادة تعيين النموذج بعد النجاح مع الإبقاء على التاريخ)

**النتيجة: 133 اختبار domain + 48 اختبار مكوّن (181 إجمالاً)، كلها ناجحة، بالإضافة
لاختبارات قواعد الأمان الناجحة عبر CI.**

## ⚠️ فجوات معروفة (موثّقة عمداً، مش أخطاء مخفية)

- **"نسخ رابط المتابعة"** بيولّد الرابط لكن مبيحدّثش `publicStats` (قواعد Firestore
  بتمنع كتابة العميل المباشرة عمداً) — محتاج Cloud Function (المرحلة 5)
- **البحث في السجل** بيدوّر بس في الجلسات المحمّلة حالياً (40 افتراضياً)
- **رسالة واتساب بعد الحفظ** لم تُنقل بعد — التطبيق الحي بيفتح واتساب تلقائياً بعد
  حفظ الجلسة؛ ميزة منفصلة نسبياً، مؤجلة لتركيز الجهد على الحفظ نفسه أولاً
- **مزامنة Google Sheets لم تُنقل عمداً** — التطبيق الحي بيبعت كل جلسة لـ Google
  Apps Script بسرّ مشترك مكتوب في كود العميل (نفس المشكلة الأمنية اللي حددناها
  في التحليل الأول). النقل الصحيح لازم يكون عبر Cloud Function (المرحلة 5)، مش
  تكرار نفس الثغرة في v2
- **"✅ تسجيل حضور جماعي"** (نافذة تسجيل حضور دفعة واحدة لكل الطلاب) لم تُبنَ بعد
- **"🌟 نجوم الحضور"** (بطاقة مشاركة صورة) لم تُنقل بعد
- **تعديل جلسة من السجل** لسه بيعرض توست "قريباً" — ✏️ في شاشة السجل بيفتح شاشة
  تسجيل فاضية بدل ما يحمّل بيانات الجلسة المطلوب تعديلها؛ يحتاج آلية تمرير حالة
  بين الشاشتين (مش موجودة حالياً، الشاشات مستقلة عن بعض)
- رابط "المتابعة" حالياً بيشاور على `child.html` الإنتاجي (RTDB)، مش بوابة v2

## تشغيل محلي

```bash
npm install
npm run dev                # خادم تطوير
npm run build               # بناء إنتاجي (يتحقق من TypeScript أولاً)
npm test                    # اختبارات domain الصافية — 133 اختبار
npm run test:components     # اختبارات المكوّنات — 48 اختبار
npm run test:rules          # اختبارات قواعد Firestore (يحتاج إنترنت عادي + Java 21+)
npm run typecheck           # فحص أنواع src/
```
